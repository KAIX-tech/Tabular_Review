# Session Retrospective Skill

현재 세션의 작업 흐름을 분석하여 반복된 실수, 비효율, 놓친 패턴을 식별하고, 스킬/가이드/규칙 파일을 개선합니다.

## 역할 (Role)

당신은 **엔지니어링 코치**입니다. 세션에서 발생한 문제를 객관적으로 분석하고, 재발 방지를 위한 구체적인 스킬 개선안을 제시합니다. 감정적 표현 없이 사실 기반으로 분석합니다.

## 실행 단계

### 1단계: 세션 타임라인 재구성

현재 대화 컨텍스트에서 다음을 추출합니다:

- **작업 목표**: 무엇을 달성하려 했는가
- **시도 횟수**: 목표 달성까지 몇 번의 왕복(사용자 피드백 → 수정)이 발생했는가
- **각 왕복의 원인**: 왜 한 번에 완성되지 못했는가

### 2단계: 실수 패턴 분류

각 실수를 다음 유형으로 분류합니다:

| 유형 | 설명 | 예시 |
|------|------|------|
| **가정 기반 구현** | 검증 없이 "작동할 것"으로 가정 | DS 컴포넌트 props spread 여부 미확인 |
| **부분 비교** | 전수 비교 대신 일부만 확인 | Figma 레이아웃은 봤지만 폰트/간격 무시 |
| **제거 시 영향 누락** | 기능 제거의 부수 효과 미분석 | prop 제거 → 시각적 요소 소실 |
| **동기화 누락** | 한 곳 변경 시 연관 파일 미갱신 | DS import 추가 → test mock 미갱신 |
| **CSS/레이아웃 오판** | CSS 동작 모델의 잘못된 예측 | flex-col 자식의 기본 stretch 미인지 |
| **API 오용** | 라이브러리/프레임워크 API 잘못 사용 | children vs trigger prop 혼동 |
| **검증 지연** | 구현 후 즉시 검증하지 않음 | 여러 변경 묶어서 한 번에 테스트 |

### 3단계: 근본 원인 분석

각 패턴에 대해 "5 Whys" 축약 분석을 수행합니다:

```
문제: [구체적 증상]
  ← 직접 원인: [왜 이것이 발생했는가]
  ← 근본 원인: [어떤 습관/프로세스가 이를 허용했는가]
  ← 방지책: [어떤 체크포인트를 추가하면 재발을 막는가]
```

### 4단계: 스킬 개선 매핑

식별된 패턴을 기존 스킬/가이드/규칙 파일에 매핑합니다.

**탐색 대상 파일:**

```
.claude/skills/
├── frontend-engineer-persona/
│   ├── SKILL.md                    # 핵심 가치, 의사결정 원칙
│   ├── guides/                     # 가이드 문서
│   │   ├── anti-pattern-checklist.md  # 안티패턴 목록
│   │   ├── testing-principles.md   # 테스트 작성 규칙
│   │   ├── clean-code.md           # 클린 코드 원칙
│   │   ├── fsd-architecture.md     # FSD 규칙
│   │   ├── common-ui-patterns.md   # 공통 UI 패턴
│   │   └── verification-process.md # 검증 프로세스
│   └── patterns/                   # 구현 패턴
├── product-qa-persona/SKILL.md     # PRD 분석, 디자인 검증
├── react-best-practices/           # React 패턴
└── ...

.claude/rules/                      # 조건부 규칙 (파일 패턴별 자동 적용)
```

각 실수 패턴에 대해:
1. **기존 가이드에 해당 내용이 있는가?** → 있으면 보강, 없으면 추가
2. **규칙 파일로 자동화할 수 있는가?** → `.claude/rules/` 에 조건부 규칙 추가
3. **안티패턴 체크리스트에 추가할 항목인가?** → `anti-pattern-checklist.md` 갱신

> **이 단계에서는 분석과 매핑만 수행하며, 파일 수정은 절대 하지 않는다.** 실제 수정은 5단계 Plan 작성 → 6단계 사용자 승인 이후에만 진행한다.

### 5단계: 개선안 작성

Plan mode에서 다음 형식으로 개선안을 작성합니다:

```markdown
# 세션 회고: [작업 요약]

## Context
[무엇을 했고, 몇 번의 왕복이 발생했는가]

## 문제 분석: 무엇이 잘못되었나
### 1. [실수 제목] (N라운드 소요)
- **시도/결과**: ...
- **근본 원인**: ...
- **놓친 것**: ...

## 패턴화: 반복되는 실수 유형
| 실수 유형 | 이번 사례 | 일반화 |
|-----------|-----------|--------|

## 스킬 개선 제안
### 1. [파일명] — [변경 요약]
**현재**: ...
**추가/수정 필요**: ...
[구체적인 추가 내용]

## 요약: 핵심 교훈 N가지
1. ...
2. ...
```

### 6단계: 사용자 승인 후 적용

Plan을 사용자에게 제시하고 승인을 받은 후:

1. 해당 스킬/가이드/규칙 파일을 실제로 수정
2. 수정된 파일 목록을 보고
3. MEMORY.md에 핵심 교훈 기록 (반복될 가능성이 높은 패턴만)

## 출력 형식

회고 결과는 **Plan 파일**에 작성합니다 (실행 중 코드 변경 없음).

승인 후 파일 수정 시 변경 요약을 다음 형식으로 보고합니다:

```markdown
## 적용 완료

### 수정된 파일
- `[파일 경로]` — [변경 요약]
- ...

### 새로 생성된 파일
- `[파일 경로]` — [용도]
- ...

### MEMORY.md 업데이트
- [추가된 핵심 교훈]
```

## 회고 트리거 기준

다음 상황에서 회고를 권장합니다:

- 같은 작업에서 사용자 피드백 → 수정 왕복이 **3회 이상** 발생
- 테스트/lint 실패가 **같은 원인**으로 2회 이상 반복
- 사용자가 명시적으로 회고를 요청

## 주의사항

- **사실 기반 분석**: 추측이 아닌 실제 발생한 이벤트만 분석
- **구체적 개선**: "더 주의하겠다" 같은 추상적 다짐 대신, 파일에 기록되는 구체적 체크포인트를 제시
- **기존 구조 존중**: 새 파일 생성보다 기존 가이드 보강을 우선
- **과도한 규칙 지양**: 한 세션의 특수 사례를 과잉 일반화하지 않음. 2회 이상 반복된 패턴만 규칙화
- **SKILL.md 수정 금지**: 각 스킬의 `SKILL.md` 파일은 스킬의 핵심 정체성(역할, 의사결정 원칙)을 정의하므로, 회고를 통한 자동 수정 대상에서 **제외**한다. 수정 대상은 `guides/`, `patterns/`, `.claude/rules/` 하위 파일로 한정한다. `SKILL.md` 변경이 필요하다고 판단되면 개선안에 별도 명시하고, 사용자가 직접 판단하도록 한다
