## 1. 제품 개요

**법률사무소의 문서들을 업로드해 구조화된 지식베이스로 만들고(DB화), Agent Chat이 그
카탈로그(DB→컬럼→문서/셀/청크)를 agentic하게 탐색해 출처와 함께 답변하는 시스템.**
화면 구성과 우선순위도 이 정의를 따른다.

- **일반 유저 — 제품의 1차 표면**: **챗 화면이 메인**이다. 자연어로 질문하면 에이전트가
  카탈로그를 단계적으로 탐색하고(진행 과정 라이브 표시), 답변에 **출처(원문 청크 또는 추출
  셀)** 를 붙인다. 근거는 **읽기전용 그리드/문서 뷰어**로 확인한다. (추출·컬럼편집 불가)
- **Admin — DB화 공정**: 워크스페이스(도메인=문서종류)를 만들어 문서를 업로드하고, 추출
  컬럼을 정의하며, 추출(Run Analysis)을 실행하고 결과를 검증한다. 즉 **Agent가 조회할 정형
  데이터를 만들고 품질을 보증하는 도구**다. 이 그리드 기능의 이름이 "Tabular Review"다.

### 사용자 역할 요약

| | 일반 유저 | Admin |
|---|---|---|
| 챗 | ✅ **메인** | ✅ |
| 문서 검색 | ✅ | ✅ |
| 그리드 보기 | ✅ **읽기전용** | ✅ 편집 가능 |
| 셀 검증/편집 | ❌ (보기만) | ✅ |
| 워크스페이스 생성/삭제 | ❌ | ✅ |
| 문서 업로드/삭제 | ❌ | ✅ |
| 컬럼 정의/편집 | ❌ | ✅ |
| 추출 실행(Run) | ❌ | ✅ |

## 2. 정보구조 (IA) — 평면 구조

워크스페이스 하나 = **하나의 도메인 = 하나의 문서종류**(= 서버 모델의 `DocumentDB`,
[domain-design.md](domain-design.md) §2.2). 워크스페이스 안에 문서 N개 × 컬럼 M개의
Tabular Review 그리드가 있다.

```
앱
├─ [User 모드]  ← 제품 기본 랜딩
│   ├─ U-0  챗 메인 (Agentic Search 대화 + 세션)     ← 일반 유저 진입점
│   ├─ U-1  검색 결과 (읽기전용 그리드)              ── 챗에서 전환/병렬
│   └─ U-2  문서 뷰어 / 인용 확인                    ── 사이드바/오버레이
│
└─ [Admin 모드]  — DB화 공정
    ├─ A-0  워크스페이스 목록 (도메인 카드)
    └─ A-1  워크스페이스 상세 = Tabular Review 그리드
             ├─ A-2  컬럼 관리 (추가/편집/라이브러리)  ── 패널/모달
             ├─ A-3  문서 업로드/관리                 ── 드롭존/패널
             └─ A-4  검증 사이드바 (셀 인용 확인)       ── 우측 사이드바
```

> 평면 구조이므로 **좌측 레일에서 워크스페이스(도메인)를 전환**하는 것이 핵심 내비게이션이다. 별도 "문서종류" 하위 단계는 없다.

## 3. 글로벌 레이아웃

두 모드 공통으로 **좌측 레일(워크스페이스 전환) + 중앙 작업영역 + 우측 사이드바(컨텍스트)** 3분할을 기본 골격으로 한다.

```
┌──────┬───────────────────────────────────────────────┬──────────────┐
│      │  Top Bar (워크스페이스명 / 액션 / 모드별 버튼)      │              │
│ 좌측 ├───────────────────────────────────────────────┤  우측         │
│ 레일 │                                               │  사이드바      │
│      │            중앙 작업영역                         │  (검증/챗/    │
│ (WS  │     (챗 또는 그리드 또는 검색결과)                 │   문서뷰어)   │
│ 전환)│                                               │              │
└──────┴───────────────────────────────────────────────┴──────────────┘
```

- **좌측 레일**: 워크스페이스(도메인) 목록 아이콘/이름. 현재 선택 강조. Admin은 하단에 "＋ 새 워크스페이스". (`domains/document-db/ui/DocumentDbRail.tsx`)
- **우측 사이드바**: 검증/출처/문서뷰어 컨테이너(400px↔900px 확장) 재사용.

## 4. USER 화면군 (일반 유저 — 제품의 1차 표면)

### U-0. 챗 메인 (진입점)

일반 유저의 기본 화면이자 **제품의 핵심 화면**. 좌측에서 도메인(워크스페이스)을 선택하고
(또는 전역) 가운데서 대화한다. 챗은 단발 RAG가 아니라 **Agentic Search**
(domain-design.md §2.13) — 에이전트가 도구를 단계적으로 호출하며 카탈로그를 탐색한다.

```
┌──────────┬──────────────────────────────────────────────┬───────────┐
│ 도메인     │  계약서에 대해 질문하기              [그리드 보기]   │           │
│ ────────  │  ──────────────────────────────────────────  │  출처 패널  │
│ ▸계약서 ●  │                                              │ (질문 후    │
│  약관     │   🙋 MFN 조항이 가장 유리한 계약은?               │  표시)     │
│  소송서면  │                                              │           │
│ ────────  │   ⏳ ① DB 탐색 중… ② 컬럼 확인 중… ③ 셀 조회 중…  │           │
│ 대화기록   │      ← 에이전트 스텝 라이브(SSE), 완료 후 접힘      │ ┌───────┐ │
│ ·오늘      │                                              │ │ACME p12│ │
│  └ MFN..  │   🤖 ACME_MSA 계약이 가장 유리합니다. 18.2조에서… │ │인용문…  │ │
│ ·어제      │      [출처 1: ACME_MSA p.12]  ← 청크(원문) 출처  │ └───────┘ │
│  └ 지배법.. │      [출처 2: MFN조항 = "있음"] ← 셀(정형) 출처   │           │
│           │  ┌────────────────────────────────────────┐  │           │
│           │  │ 계약서에 대해 무엇이든 물어보세요…       [↑]│  │           │
│           │  └────────────────────────────────────────┘  │           │
└──────────┴──────────────────────────────────────────────┴───────────┘
```

- **좌측**: 도메인(워크스페이스) 목록 — 유저는 **읽기 전용으로 선택만**(선택 = 세션 스코프,
  미선택 = 전역으로 에이전트가 DB를 직접 고름). 하단에 대화 기록(세션 목록, 서버 영속).
- **중앙 챗**:
  - **에이전트 스텝 라이브 표시**: 메시지 전송 후 SSE `step` 이벤트("DB 탐색 중…", "컬럼
    확인 중…", "셀 조회 중…")를 타임라인으로 표시. 답변 완료 후엔 접힌 "탐색 과정"으로
    보존 — 세션 재오픈 시 `ChatMessage.steps`로 재현(phase-4 §2.2, §4).
  - **출처 칩 이원화**: 청크 출처(①, 문서명+페이지 → 문서 뷰어 U-2로 점프) / 셀 출처(②,
    컬럼명=값 → 읽기전용 그리드 U-1의 해당 셀로 점프). `.tr-cite` 번호형 칩.
  - 스트리밍 미지원 환경은 최종 답변만 받는 비스트리밍 폴백(phase-4 §4).
- **상단 `그리드 보기`**: 현재 선택 도메인의 읽기전용 그리드(U-1)로 전환.
- **빈 상태**: 추천 질문 칩 ("이 도메인에서 가장 흔한 조항은?", "계약일이 빠른 순서는?").
- **입력**: 멀티라인, 전송(↑).
- **코드 매핑**: `domains/chat/ui/ChatMainPage.tsx` + `ChatInterface.tsx` +
  `ChatSessionList.tsx`. 세션은 현재 localStorage → phase-4에서 서버 영속 전환.

### U-1. 검색 결과 (읽기전용 그리드)

일반 유저가 "그리드 보기"를 누르거나 셀 출처 칩에서 점프했을 때. **A-1과 동일 레이아웃이나
편집 UI 전부 제거**.

```
┌──────────┬──────────────────────────────────────────────┬───────────┐
│ 도메인     │ 계약서   [🔍 검색…           ]        [챗으로]    │           │
│ ▸계약서 ●  │ ─────────────────────────────────────────────  │  문서 뷰어  │
│           │  문서명        │ 지배법   │ 계약일   │ MFN조항      │ (셀 클릭)  │
│           │ ──────────────┼────────┼────────┼──────────     │           │
│           │  ACME_MSA.pdf │ 뉴욕주   │ 2024-01 │ 있음          │           │
│           │  Beta_NDA.pdf │ 델라웨어 │ 2023-11 │ 없음          │           │
└──────────┴──────────────────────────────────────────────┴───────────┘
```

- A-1 그리드 재사용하되 **숨김**: 체크박스, `＋컬럼`, 컬럼 편집 진입, Run/Stop, 문서 추가/삭제, 검증 버튼.
- **추가**: 상단 검색바(문서명·셀값 필터, 후속 단계에서 의미검색으로 확장), 정렬/필터(읽기),
  챗 셀 출처에서 점프 시 해당 셀 하이라이트.
- 셀 클릭 → **읽기전용 인용 사이드바**(A-4에서 `검증 완료` 버튼만 제거한 형태) / 문서 뷰어(U-2).

## 5. ADMIN 화면군 (DB화 공정)

Admin 화면군은 **Agent가 조회할 카탈로그를 만드는 공정**이다 — 문서 업로드(인제스천) →
추출 스키마(컬럼) 정의 → 추출 실행 → 사람 검증. 산출물(문서/청크/셀)이 곧 챗 에이전트의
탐색 대상이 된다.

### A-0. 워크스페이스 목록 (도메인 대시보드)

Admin 진입 시 첫 화면. 도메인=문서종류 카드 그리드.

```
┌──────┬────────────────────────────────────────────────────────────┐
│  ▦   │  Workspaces                              [＋ 새 워크스페이스]  │
│  WS  │  ──────────────────────────────────────────────────────────  │
│ 레일 │  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐         │
│      │  │ 📄 계약서       │ │ 📄 약관        │ │ 📄 소송서면     │         │
│ ▦계약│  │ 문서 128 · 컬럼 9│ │ 문서 54 · 컬럼 6│ │ 문서 12 · 컬럼 4│         │
│ ▦약관│  │ 최종수정 6/2    │ │ 최종수정 5/28  │ │ 최종수정 5/19  │         │
│ ▦소송│  │ [열기] [⚙][🗑] │ │ [열기] [⚙][🗑] │ │ [열기] [⚙][🗑] │         │
│  ＋  │  └──────────────┘ └──────────────┘ └──────────────┘         │
└──────┴────────────────────────────────────────────────────────────┘
```

- **카드 구성요소**: 도메인 아이콘/이름, 문서 수, 컬럼 수, 최종수정일, 진행상태(예: 추출 완료율 바).
- **액션**: `열기`(→ A-1), `⚙ 설정`(이름/문서종류·아이콘 변경), `🗑 삭제`(확인 모달).
- **＋ 새 워크스페이스**: 모달 — 이름, 문서종류 라벨, (옵션) 컬럼 템플릿 프리셋 선택.
- **빈 상태**: "아직 워크스페이스가 없습니다. 첫 도메인을 만들어 문서를 추가하세요." + CTA.
- **코드 매핑**: `domains/document-db/ui/DocumentDbListPage.tsx`.

### A-1. 워크스페이스 상세 = Tabular Review 그리드

```
┌──────┬───────────────────────────────────────────────────────┬─────────┐
│  ▦   │ 계약서  ▸ Untitled Project   [Chat][Export][⊞Wrap]      │         │
│  WS  │                       [＋문서][모델▾][▶ Run Analysis]    │  우측    │
│ 레일 ├───────────────────────────────────────────────────────┤ 사이드바 │
│      │ ☐│ 문서명        │ 지배법    │ 계약일    │ MFN조항  │ ＋   │ (verify │
│ ▦계약│ ─┼──────────────┼─────────┼─────────┼────────┼────  │  /none) │
│ ▦약관│ ☐│ ACME_MSA.pdf │ 뉴욕주 ●  │ 2024-01 ●│ 있음 ⚠  │      │         │
│ ▦소송│ ☐│ Beta_NDA.pdf │ 델라웨어●  │ 2023-11 ●│ 없음 ●  │      │         │
│      │ ☐│ ...          │ extracting…(스피너)            │      │         │
│  ＋  │                                                       │         │
└──────┴───────────────────────────────────────────────────────┴─────────┘
```

- **상단 바**: 워크스페이스명(도메인) + 프로젝트명(인라인 편집), 우측에 액션군 — `Chat`, `Export CSV`, `Wrap`, `＋문서`, `모델 선택`, `Run Analysis`(또는 선택 행 있으면 `Re-run (N)`, 처리중이면 `Stop`).
- **그리드(중앙)**:
  - 행 = 문서, 열 = 컬럼. 좌측 체크박스로 행 선택(선택 재실행용).
  - 셀 = 추출값 + 신뢰도 점(High/Med/Low 색상) + 상태 아이콘(✓검증됨 / ⚠Low확인필요)
    + 추출 방식 구분(retrieval_fallback 셀엔 "부분 컨텍스트" 배지 — domain-design.md §2.12).
  - 열 헤더: 컬럼명 + 타입, 헤더 클릭 → 편집(A-2), 우측 끝 `＋`로 컬럼 추가.
  - 셀 클릭 → 우측 검증 사이드바(A-4). 문서명 클릭 → 문서 뷰어.
  - 드래그&드롭으로 문서 추가.
- **상태 표시**: 컬럼 추출 중 스피너, 변환 중 문서별 진행 표시.
- **코드 매핑**: `domains/document-db/ui/DocumentDbReviewPage.tsx` +
  `domains/document-review/ui/DataGrid.tsx`.

### A-2. 컬럼 관리 (추가 / 편집 / 라이브러리)

기존 `domains/document-review/ui/AddColumnMenu.tsx` + `ColumnLibrary.tsx` 대응. 그리드 위 팝오버/모달.

```
컬럼 추가/편집 팝오버                      컬럼 라이브러리 모달
┌─────────────────────────┐            ┌──────────────────────────────┐
│ 컬럼 편집            [라이브러리]│            │ Column Library        [Import]│
│ 이름  [ 지배법          ]│            │ 카테고리: [전체][법률][재무][날짜]│
│ 타입  [text ▾]          │            │ ┌───────────────┐            │
│ 프롬프트                  │            │ │ 지배법 (text)    │ [＋추가]   │
│ [ What is the governing │            │ │ "What is the..." │           │
│   law of this contract?]│            │ ├───────────────┤            │
│        [✨ AI로 프롬프트 생성]│            │ │ 계약 총액 (number)│ [＋추가]   │
│ [삭제]          [취소][저장]│            │ └───────────────┘            │
└─────────────────────────┘            └──────────────────────────────┘
```

- **컬럼 편집**: 이름 / 타입(text·number·date·boolean·list·select 계열) / 자연어 프롬프트 / `AI 프롬프트 생성` / 삭제.
- **라이브러리**: 저장된 템플릿을 카테고리별로 탐색 → 클릭 추가. Import/Export(localStorage 기반, `domains/document-review/lib/column-library.ts`).
- **Admin 전용** — 일반 유저에게는 컬럼 헤더의 `＋`/편집 진입점 자체를 숨긴다.
- 컬럼의 prompt가 곧 챗 에이전트가 `list_columns`로 읽는 추출 스키마 설명이 된다 — 좋은
  프롬프트는 추출 품질과 에이전트의 컬럼 선택 정확도를 함께 올린다.

### A-4. 검증 사이드바 (셀 인용 확인)

기존 `domains/document-review/ui/VerificationSidebar.tsx`. 셀 클릭 시 우측에서 확장.

```
┌─────────────────────────────────┐
│ 지배법                       [확장][✕]│
│ ─────────────────────────────── │
│ 값:  뉴욕주 (New York)            │
│ 신뢰도: ● High                   │
│ 추론:  계약서 18.2조에 명시…         │

│ ─────────────────────────────── │
│ …governed by the laws of the    │
│ State of [New York]…(하이라이트)   │  ← 확장 시 원문 전체 + <mark>
│ ─────────────────────────────── │
│           [✓ 검증 완료]            │
└─────────────────────────────────┘
```

- 좁게(400px): 값/신뢰도/추론/출처 요약. 넓게(900px): 원문 전체에서 인용구 노란 하이라이트 + 자동 스크롤.
- `검증 완료` → 셀 `review_status: verified`(✓). **이 검증 액션은 Admin 전용.**
- 검증된 값은 챗 에이전트가 `query_cells`로 인용하는 정형 데이터의 신뢰 기반이 된다.

## 8. 라우팅 · FSD · 상태 매핑 (Next.js App Router)

### 라우트 맵 (현행)

| 화면 | 라우트 | 비고 |

> 당초 `(admin)`/`(user)` route group 분리안은 **단일 `(shell)` 그룹 + 역할별 UI 노출
> 제어**로 수렴했다(인증이 범위 밖이라 라우트 차원의 역할 분리는 실익이 없음). 인증/권한
> 도입 시 route group 분리를 재검토한다.

`app/`는 라우팅 전용 — 각 `page.tsx`는 도메인의 페이지 컴포넌트만 import한다.

### FSD 배치
- `domains/document-db` — 워크스페이스(DocumentDB) 목록·카드·좌측 레일·전환 상태 + 그리드 페이지(`DocumentDbListPage`, `DocumentDbReviewPage`).
- `domains/document-review` — 그리드/컬럼/검증/업로드/뷰어 (Tabular Review 기능군).
- `domains/chat` — 챗 메인/인터페이스/세션 + 출처 칩.
- `widgets/` — 도메인을 조합하지 **않는** 순수 레이아웃(3분할 셸 등). 도메인 Public API를 조합하면 widget이 아니라 해당 도메인에 둔다(이 스킬 FSD에선 widget→domain import 금지).
- `shared/ui` — 공통 프리미티브(빈 상태, 카드, 사이드바 셸 등).

### 상태 · 데이터 레이어
- **Zustand** (Provider + selector) — 선택된 워크스페이스, 사이드바 모드, 역할(admin/user) 등 클라이언트 UI 상태. `domains/*/model`.
- **TanStack Query** — 변환/추출/챗/워크스페이스/검색 등 서버 호출. mock/real은 도메인별 토글로 전환(§9).

## 9. 목데이터(Mock) → Real 전환 전략

UI/UX를 mock으로 먼저 검증하고, 백엔드 컨텍스트가 구현되는 대로 도메인별로 전환한다.
경계는 API 함수(`*.api.ts`)에 모여 있어 화면 코드 무변경으로 교체된다.

### 전환 현황

| 도메인 | mock 토글 | 상태 |
|---|---|---|
| DocumentDB/컬럼 | `NEXT_PUBLIC_USE_MOCKS_DOCUMENT_DB` | 백엔드 구현됨 → real 전환 |
| 문서 인제스천/그리드/추출 | `NEXT_PUBLIC_USE_MOCKS_REVIEW` | 백엔드 구현됨 → real 전환 |
| 챗 | `NEXT_PUBLIC_USE_MOCKS_CHAT` | **mock — phase-4(챗 컨텍스트)에서 real 전환** |

- **단일 글로벌 + 도메인별 오버라이드**: `NEXT_PUBLIC_USE_MOCKS`(글로벌 기본) + 도메인별
  플래그(`shared/config/env.ts`의 `ENV.mocks.*`). 컨텍스트가 구현될 때마다 도메인 단위로 전환.
- **경계 고정**: mock은 실제 API와 **동일한 타입·시그니처**를 반환 → 백엔드 준비 시 함수 본체만 교체, 화면 코드는 무변경.
- **지연·에러 시뮬레이션**: mock에 인위적 지연/실패를 넣어 로딩·에러·빈 상태(§6)를 실제로 확인.
